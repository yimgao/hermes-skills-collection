# arXiv Paper Summarizer

> Summarize academic papers from arXiv — extracts the key contribution, method, results, and implications into a structured, readable summary in English or Chinese. Optionally monitors arXiv for new papers on a topic.

---

## Overview

This skill takes an arXiv paper link (or a search query) and produces a clean, structured summary. It reads the abstract and full paper content, extracts the core contributions, methodology, experimental results, and limitations — then compiles everything into a digestible format.

Perfect for staying current with research, writing literature reviews, or quickly deciding whether a paper is worth reading in full.

**What it handles:**
| Input | Example |
|-------|---------|
| **Single paper URL** | `https://arxiv.org/abs/2401.12345` |
| **arXiv ID** | `2401.12345` |
| **Search query** | *"Latest papers on LLM reasoning from 2026"* |
| **Batch** | *"Summarize these 5 papers about RAG"* |

**Output languages:**
- English (default)
- 中文 (Chinese)
- Bilingual (English + Chinese side by side)

## Core Workflow

### Step 1: Fetch Paper Data

Given an arXiv URL or ID, fetch metadata and full text:

#### 1a. Get Metadata (Abstract API)
```bash
curl -s "https://export.arxiv.org/api/query?id_list={paper_id}" 2>&1
```

This returns an XML response with:
- **Title**
- **Authors** (full list)
- **Abstract**
- **Published date**
- **Updated date**
- **Categories** (cs.AI, cs.CL, etc.)
- **DOI / comment** (if available)
- **Primary category**

#### 1b. Get Full PDF / HTML
If the user wants a deeper summary (beyond the abstract), fetch the paper's full content:

```
HTML abstract page: https://arxiv.org/abs/{paper_id}
PDF: https://arxiv.org/pdf/{paper_id}
```

For HTML, parse the page content to extract:
- Section headings
- Figures / tables (if mentioned)
- Conclusion section

For PDF, use available tools to extract text.

### Step 2: Analyze Paper

Read and analyze the paper's content:

**Key questions to answer:**
```
1. What problem does this paper solve?
2. What is the proposed method / contribution?
3. How does it compare to prior work?
4. What experiments were conducted?
5. What are the main results (quantitative)?
6. What are the limitations?
7. What are the broader implications?
8. Who should care about this (target audience)?
```

Use the following template (adapt language per user's request):

### Step 4: Handle Search Queries

If the user provides a search query instead of a paper ID:

```
"{topic} arxiv {year}"
"{topic} site:arxiv.org"
```

Then:
1. Search for recent papers on the topic
2. Select the most relevant 3-5 papers
3. Summarize each paper using the template above
4. Optionally compile a comparison table:

```markdown
## 📋 Paper Comparison

| Paper | Year | Method | Key Result | Code? |
|-------|------|--------|-----------|-------|
| {Title} | 2026 | {Method} | {Result} | ✅/❌ |
| {Title} | 2026 | {Method} | {Result} | ✅/❌ |
```

### Step 5: Save & Offer Cron (for search monitoring)

Save individual summaries:

```
~/research/arxiv/{category}/{paper_id}-summary.md
```

Save search result comparisons:

```
~/research/arxiv/searches/{topic}-{YYYY-MM-DD}.md
```

If the user wants ongoing monitoring for new papers:

```bash
cronjob \
  action=create \
  schedule="0 8 * * 1" \
  name="arxiv-monitor-{topic}" \
  prompt="Search arXiv for new papers on {topic} from the past week. Summarize the top 3 most interesting papers."
  skills=["arxiv-paper-summarizer"]
```

## Common Pitfalls

| # | Problem | Solution |
|---|---------|----------|
| 1 | **arXiv API returns XML.** | Parse properly with XML tools. Extract from `<entry>` elements. |
| 2 | **Abstract is too short for deep analysis.** | Fetch the full HTML page for more context. Read the conclusion section. |
| 3 | **PDF extraction is messy.** | Prefer the HTML abstract page + full-text HTML if available. PDF as fallback. |
| 4 | **Multiple authors with affiliations.** | List authors only. Affiliations are secondary detail. |
| 5 | **arXiv IDs change format.** | Older IDs use `YYMM.NNNN` format; newer ones use `YYMM.NNNNN`. Handle both. |
| 6 | **Results are too technical.** | Translate into conceptual terms. Focus on "what" not "how" for non-expert summaries. |
| 7 | **Paper has no code / project page.** | Note as "not available". Don't fabricate links. |
| 8 | **中文翻译失去技术精确性。** | Keep technical terms in English with Chinese explanation. E.g., "chain-of-thought (思维链) reasoning". |

## arXiv API Reference

```
# Fetch by ID
https://export.arxiv.org/api/query?id_list={id1},{id2}

# Search
https://export.arxiv.org/api/query?search_query=all:{query}&start=0&max_results=10

# Common search prefixes
all:        Everything (title + abstract + author)
ti:         Title only
au:         Author
abs:        Abstract only
cat:        Category (e.g., cat:cs.AI)
```

**Important:** arXiv API rate limits apply. Batch requests with `&max_results=`. Use `&sortBy=submittedDate&sortOrder=descending` for recent papers.

## 📋 Data Sources & Accuracy

This skill relies on searching external data. Follow these guidelines to ensure accuracy:

### Primary vs Secondary Sources
| Priority | Source Type | Examples |
|----------|-------------|----------|
| 🥇 **Primary (prefer)** | Government / Official | Census.gov, BLS, city economic dev, official APIs |
| 🥈 **Secondary** | Aggregators / Reviews | Google Maps, Yelp, LoopNet, Wikipedia |
| 🥉 **Tertiary** | Estimates / Inferences | Industry reports, blog posts, news articles |

### Accuracy Rules
1. **Always cite the source** for each data point — don't state numbers without attribution
2. **When sources conflict**, prefer: government > official > aggregator > blog
3. **If data is unavailable**, say "Not found" — do NOT fabricate or approximate without stating it's an estimate
4. **Mark estimates clearly** with "~" or "estimated" — never present guesses as facts
5. **Use ranges** when uncertain (e.g., "$20-25/sqft" not "$22.50/sqft")
6. **Timestamp your data** — note when it was collected (e.g., "as of June 2026")

### When Web Search Returns Limited Data
- Note the limitation explicitly: "Google Maps limited view", "Search blocked by CAPTCHA"
- Fall back to known data or general market knowledge
- Explicitly mark knowledge-based data as "based on general market knowledge"
- Never invent competitor names, ratings, or prices

### Verification Checklist
- [ ] Every number attributed to a source
- [ ] Estimates clearly marked (use "~" or "estimated")
- [ ] Unavailable data noted as "not found"
- [ ] Data collection timestamp recorded
- [ ] Confidence level noted (High / Medium / Low)
